# JTEST Marketing Analytics Operating Knowledge v1.0（日本語版）

**ドキュメント種別：** オペレーティングナレッジ（実装報告書ではない）
**目的：** 現在のマーケティング分析アーキテクチャがなぜ構築されたのか、そして今後どのように運用すべきかを説明する
**想定読者：** 将来のプロジェクトオーナー、および本作業の記憶を持たない将来のAIアシスタント
**ステータス：** Active — Phase 2-A完了時点の状態を反映
**スタック：** Facebook Ads + Lovable + GitHub + GA4

---

## 1. 背景

この取り組みを始める前、J.TEST Navi Vietnamのマーケティング計測は広告プラットフォームの境界線で止まっていた。Facebook Adsは広告自体のクリック数とインプレッション数を報告できたが、ユーザーがアプリに到達した後に何が起きたかについては何も分からなかった。「広告をクリックした人のうち何人が実際にJ.TEST Naviを開いたのか」「何人がゴールを選択するなど実際に利用したのか」「何人が診断を完了したのか」「何人が次の実質的なステップであるPreCheckの確認に進んだのか」といった基本的な問いに答えることができなかった。

これはよくあることだが、深刻なギャップである。広告プラットフォームが計測するのは広告のパフォーマンスであり、プロダクトへのエンゲージメントではない。この両者をつなぐ橋がなければ、「広告の成果は良い」という主張と「その広告は実際にプロダクトを使う人を連れてきている」という主張は、まったく別の、検証されていない主張のままである。クリック数・インプレッション数だけに基づく予算判断は、実質的には広告の先で何が起きているかについての推測に過ぎない。

この取り組みの目的は、このギャップを埋めることだった。すなわち、新しいインフラを導入するのではなく、すでに存在するツール（フロントエンドとしてのLovable、そのソース管理としてのGitHub、計測レイヤーとしてのGA4）を使って、広告クリックから、本当の関心を示す意味のあるプロダクト内アクションまでを一貫してつなぐ、計測可能なマーケティングファネルを構築することである。

---

## 2. 現在のマーケティングファネル

現在、計測可能なファネルは以下の通りである。

```
Facebook Advertisement
  ↓
J.TEST Navi (Lovable)
  ↓
goal_selected
  ↓
diagnosis_completed
  ↓
precheck_clicked
```

これはこのファネルの**最初の計測可能なバージョン**である。本当に関心を持つユーザーがたどる中核的な経路を捉えている。すなわち、広告から到達し、自分が達成したいことを宣言し（ゴール選択）、診断体験を完了し、J.TEST Vietnamファネルにおける次の実質的なステップであるPreCheckへとクリックして進む、という流れである。この図の各矢印は、計画中や願望としての段階ではなく、すでに実装され、本番環境で検証済みの実際のGA4イベントに対応している。

これは意図的に、可能なファネル全体ではない。`precheck_clicked` で止まっているのは、Phase 2-Aのスコープが意図的にそこまでに限定されたためである（第4章参照）。このファネルは、今後のフェーズでさらに拡張されることが想定されている。

---

## 3. なぜGA4を導入したのか

ページビューだけでは、「誰かがページを読み込んだかどうか」という一つの問いにしか答えられない。到達した後にその人が何か意味のある行動をとったかどうかは分からない。J.TEST Naviのような診断ツールにおいては、ユーザーが実際に本物の診断を完了した場合と、すぐに離脱した場合とで、結果画面のページビューは見分けがつかない。ページビューには、意図や結果についての情報が一切含まれていない。

カスタムイベントは、ゴールの選択、診断の完了、PreCheckへのクリックといった特定のプロダクト内アクションに意味を紐づけることで、この問題を解決する。これらは、ユーザーが実際のリードへと進みつつあることを示す瞬間であり、マーケティングの意思決定は生のトラフィック量ではなく、これらに基づいて行われるべきである。

この実装全体を通じて厳格に守られてきた原則であり、今後のあらゆるフェーズにおいても引き続き守られなければならない原則がある。**分析は観察レイヤーであり、ビジネスロジックではない。** GA4のコードは、ゴールが選択された、診断結果が計算された、リンクがクリックされたといった、アプリ内ですでに起きていることを観察するだけである。これらの事象を決定したり、変更したり、ゲートしたりすることは決してあってはならない。診断計算、ルーティング、UIは、GA4が存在するか、到達可能か、あるいはユーザーのブラウザによってブロックされているかにかかわらず、まったく同じように動作しなければならない。この分離こそが、プロダクト自体を危険にさらすことなく、分析機能を安全に追加・削除・拡張できる理由である。

## 4. 現在のフェーズ

**Phase 1 — 標準ページトラッキング（完了・稼働中）**
- `page_view`
- `session_start`
- `first_visit`

**Phase 2-A — コアファネルイベント（完了・稼働中）**
- `goal_selected`
- `diagnosis_completed`
- `precheck_clicked`

**Phase 2-B — 保留中、意図的に延期**
- `fgps_clicked`
- `jtest_application_clicked`

Phase 2-Bは見落としではなく、意図的に延期されている。その理由は以下の通りである。

- **下流のプロダクトがまだ完全には整備されていない。** FGPSおよびJ.TEST申込フローは、より広いプロダクトロードマップにおける後段階のステップである（第7章参照）。まだ安定した最終的な着地点を持たないアクションに対してイベントを追加すると、分析される前にその形が変わってしまうものを計測することになりかねない。
- **段階的な展開はリスクを抑える。** これまでの各フェーズは、完全に実装し、GA4のDebugView/リアルタイムで検証し、次に進む前にLovableのRemix/Publishサイクルを経ても機能が維持されることを確認できる程度の小さな規模に保たれてきた。Phase 2-BをPhase 2-Aとまとめてしまうと、一つの変更が大きくなりすぎ、きれいに検証することが難しくなっていたはずである。
- **ファネルは、ユーザーが実際にたどる順序で拡張されるべきである。** `goal_selected` → `diagnosis_completed` → `precheck_clicked` は、直近の、すでに稼働している経路である。`fgps_clicked` と `jtest_application_clicked` は、その次のステップに属するものであり、プロダクト自体のそのステップが計測できる状態になってから追加する方が意味を持つ。

Phase 2-Bは、FGPSおよびJ.TEST申込フローの準備が整った段階で、Phase 1・Phase 2-Aで用いたのと同じ段階的・no-inference（推測禁止）の規律に従い、独立した個別スコープの実装プロンプトとして着手すべきである。

## 5. リポジトリ構成

GitHubは、アプリケーションコードと、その分析実装を説明するドキュメントの両方における唯一の正（single source of truth）である。分析アーキテクチャに関する内容は、リポジトリにコミットされていない限り、権威あるものとはみなされるべきではない。

推奨されるドキュメント構成は以下の通りである。

```
docs/
  analytics/
    Analytics_Implementation_Spec_v1.0.md
    Analytics_Event_Definition_v1.0.md
    JTEST_Marketing_Analytics_Operating_Knowledge_v1.0.md   (this document)
```

- `Analytics_Implementation_Spec_v1.0.md` は、実際に構築された技術実装を記述するものであり、`src/lib/analytics.ts` の内容、`initGA()`・`trackPageView()`・`trackEvent()` がどこから呼び出されているか、Remix/Publishを乗り越えるための注意点などを含む。
- `Analytics_Event_Definition_v1.0.md` は、現在実装されているすべてのGA4イベント、その正確なパラメータ、そしてその意味を記載する正典的なリストであるべきであり、PreCheckの問題データにおけるCanonical Test Master Schemaと同じ役割を果たす。新しいフェーズがリリースされた瞬間に更新すべきであり、後追いで更新すべきではない。
- 本ドキュメント（`JTEST_Marketing_Analytics_Operating_Knowledge_v1.0.md`）は「何を（what）」ではなく「なぜ（why）」を扱うものである。理由づけと運用上の規律を説明するものであり、仕様書・定義書に比べてはるかに変更頻度が低いはずである。

このワークフローのどこかでGoogle Driveを使用する場合、それは運用上のアウトプット——エクスポートしたレポート、その場限りの分析、GA4ダッシュボードのスクリーンショットなど——を保管する場としてのみ用いるべきであり、何が・なぜ実装されたかについての権威ある定義を保管する場としては決して用いるべきではない。Driveのドキュメントと、GitHubにコミットされたドキュメントの内容が食い違った場合は、常にGitHubが優先する。

## 6. 日常運用

Facebook Adsのパフォーマンスは、広告プラットフォームの指標のみで評価してはならない。広告パフォーマンスを確認する際の推奨手順は以下の通りである。

```
1. Facebook Ads
   - Link Click
   - CTR
   - CPC
        ↓
2. GA4
   page_view
        ↓
   goal_selected
        ↓
   diagnosis_completed
        ↓
   precheck_clicked
```

まずFacebook Adsでコストとリーチを把握する——クリックを1件獲得するのにいくらかかっているか、広告コピー・クリエイティブがどれだけ効率的にクリックを生み出しているかを確認する。その後、GA4に移り、ファネルを順番にたどる。クリックが実際にページの読み込みにつながったか、訪問者がエンゲージしたか（ゴール選択）、コア体験（診断）を完了したか、そして次の実質的なステップ（PreCheck）に進んだかを確認する。

CPCが低くても `page_view` から `goal_selected` の間で大きく離脱しているキャンペーンは、実際には安価で効率の良いキャンペーンではなく、エンゲージしない安価なトラフィックにすぎない。逆に、CPCが高くても `precheck_clicked` まで高い完了率を示すキャンペーンは、表面上のクリック単価が悪く見えても、より優れた投資である可能性がある。**予算配分やクリエイティブの意思決定は、クリック数やCPC単体ではなく、ファネル全体の形状に基づいて行われるべきである。** これこそが、このアーキテクチャを構築した根本的な理由であり、そもそもこうした比較を可能にするためのものである。

## 7. 今後の拡張

計画されている長期的なプロダクトの道筋は以下の通りである。

```
PreCheck FG
  ↓
PreCheck DE
  ↓
FGPS
  ↓
J.TEST Application
  ↓
LAOS analytics
```

J.TEST Naviのマーケティングファネル（第2章）は、このより大きな道筋への入口である。下流の各プロダクトが安定するにつれて、それぞれの主要なアクションについても、PreCheckおよびJ.TEST Naviがすでに受けてきたのと同じ扱い——明確に定義された、no-inference（推測禁止）に準拠した少数のイベントを、意図的にスコープを限定したフェーズで追加し、GA4で検証し、`Analytics_Event_Definition` に文書化するという扱い——を適用すべきである。

長期的なゴールは、この道筋上のすべてのプロダクト——PreCheck FG、PreCheck DE、FGPS、J.TEST申込フロー、そして最終的にはLAOSレベルの分析——が、一つの分析標準を共有することである。すなわち、同じ `trackEvent()` 方式のパターン、同じ `app_name`／パラメータの規律、同じ段階的展開のアプローチ、そして第8章に示す同じ運用原則である。標準を共有することで、各プロダクトが互いに互換性のない独自のトラッキング方式を発明するのではなく、すべてのプロダクトのデータを将来的に同じ土俵で比較・統合できるようになる。

## 8. 重要な設計原則

- **GitHubが正である。** アプリケーションコードと分析ドキュメントは、そこにコミットされて初めて権威あるものとなる。
- **Google Driveは運用上のアウトプットのみを保管する。** レポート、エクスポート、ダッシュボードなどであり、権威ある実装記録や定義記録は保管しない。
- **GA4は観察者であり、ビジネスロジックではない。** 分析コードが診断計算、ルーティング、UIの挙動を変更・ゲート・依存させることは決してあってはならず、失敗した場合にナビゲーションやプロダクトの機能をブロックすることも決してあってはならない。
- **分析実装を重複させない。** 初期化関数（`initGA()`）は一つ、ページトラッキング関数（`trackPageView()`）は一つ、汎用イベント関数（`trackEvent()`）は一つ、すべて一つのファイルの中に存在する。今後のすべてのフェーズはこれらを再利用する。
- **個人情報を収集しない。** 氏名、メールアドレス、電話番号その他の個人を特定できる情報を、いかなるイベントパラメータにも、決して含めない。
- **一度にすべてを実装するのではなく、フェーズを分けて拡張する。** 各フェーズは、次のフェーズに進む前に完全に実装・検証・文書化できる程度の小さなスコープに保たれている——これが、Phase 1およびPhase 2-Aを安全に構築し、本番環境で動作を確認できたことの理由である。

## 結論

このアーキテクチャがJ.TEST Vietnamにとって価値を持つ理由は、単に「データが増える」ということにとどまらない。この取り組みが行われる前、マーケティング予算の意思決定は、プロダクトが実際にその対象となる人々にとって機能しているかどうかについて何も語らないシグナル——クリック数とインプレッション数——に基づいて行われていた。このギャップのために、安価だが効果のないキャンペーンと、高価だが効果のあるキャンペーンを区別することも、J.TEST Navi自体への変更が実際のユーザーエンゲージメントを高めたのか損なったのかを知ることも、構造的に不可能だった。

Facebook Adsを `goal_selected`、`diagnosis_completed`、`precheck_clicked` までつなげることで、マーケティングの意思決定は、単に何が表示されたかではなく、ユーザーが実際に何をしたかに基づけるようになった。これにより、広告費の評価は推測から計測へと変わる。しかもそれは、プロダクト本来のビジネスロジックに一切触れることなく、それを危険にさらすこともなく実現されている——分析レイヤーは、あくまで観察するだけである。

同じくらい重要なのは、これを構築するために用いられた、段階的・no-inference（推測禁止）・単一の正の情報源という規律そのもの（小さくスコープを限定したフェーズ、データを創作しない、実装は一つでどこでも再利用する、GitHubに文書化する）が、それ自体長期的な資産だということである。これにより、Phase 2-B、PreCheck DE、FGPS、そして将来のJ.TEST申込フローのすべてを、同じ信頼できる方法で計測できるようになる。データへの信頼をその都度ゼロから築き直すのではなく、プロダクトを一つずつ追加しながらこのファネルを拡張していくことができる。長期的に見れば、J.TEST Vietnamがプロダクトラインを拡大していく中で、確信を持ってエビデンスに基づいたマーケティングの意思決定を行えるようにするのは、個々のイベントやダッシュボードではなく、この一貫性そのものである。
